Overview

Regression Testing (RT) is performed to assess & find out the potential impact the new changes could have on the existing functionalities/business processes and work with the respective teams to eliminate them before moving into Production.

What are we doing?

  • Regression Testing for all the Projects & Warranty, Enhancement Requests & Warranty and Production Support fixes across all regions that goes into Production
  • EHP/SP Upgrade Testing across SAP systems landscape as applicable
  • Maintain & own Regression Test Scripts Repository as custodians (1700+ scripts across APPLE Business processes as of Feb-2017)
  • Perform Onapsis Security Testing, iOS/MacOS Upgrade Testing and Browser Testing as required
  • Automate RT Test scripts which qualify and use for Regression Testing as well as EHP/SP upgrades
  • Perform testing for Month end Vertex patch upgrades planned for tax updates
  • Carry out detailed Change Request (CR) Analysis periodically and ascertain the root causes
  • Manage Change Management process end-to-end. Change management process ensures that new changes are moved to Production systems with built in checks & balances to have a seamless implementation
  • Maintain & own the HP ALM Tool to be used for Regression and EHP/SP upgrade Testing as well as the Test Script Repository
  • Follow RT-PLC (RT - Project Life Cycle) for all Projects. Involve and travel throughout the lifecycle of the project right from Kick-Off until P2S handover:
    • Participate in Project Kick off & Design meetings
    • Identify, share and walk through RT test scripts relevant to the change with the projects team for IT
    • Understand the changes with projects team conducting System Demo and IT test case walk through
    • Review IT & UAT Test results and defects as applicable
    • Discuss with Project team, Scope and Perform Regression testing for all regions as applicable
    • Organize retrospective meeting with the projects team and seek feedback on RT performance (In-progress)
    • Perform Root Cause Analysis for defect slippage due to RT miss, if any

Key asks from RT Team

  • Provide enough heads-up on the upcoming RT requirement well in advance for better planning and scoping (particularly when there is a change in the project timelines)
  • General RT time line of 2 weeks is subject to change upwards/downwards depending on the scope and size of the projects / tickets; this would be mutually discussed and agreed
  • RT Scoping Template completeness is mandatory for RT execution. Get more information in regression test scoping
  • Project Team to share the unique IT scripts post IT phase or during P2S handover phase, which would be inducted into the RT Test Repository post validation
  • RT team's involvement during Project Life Cycle should be an integral part of the PLC. Get more information on RT processes
  • CS-SR tickets with Catalog item "SAP Regression Testing" including details in RT scoping template has to be assigned to workgroup SAP RT Team
  • Any fix that involves configuration, variant change, batch job changes will typically not require RT to be performed. The concerned team shall work with the RT team and may get a waiver accordingly
